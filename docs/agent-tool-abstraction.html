一

前提知識

Prerequisites

抽象化の設計は、抽象化される側の具体的な事情を知らずには評価できない。 まず ccsession がどう動いているか、そして Claude Code と OpenCode が セッションをどう保存しているかを押さえる。

1.1ccsession のアーキテクチャ — 再帰サブプロセスモデル

ccsession は「claude --resume の fzf フロントエンド」で、実体は1つの Go バイナリに 4つの顔がある。

  • ccsession list — セッション一覧を TSV 行として標準出力に吐く
  • ccsession preview <id> — 1セッションのプレビュー（直近30メッセージ）を描画する
  • ccsession resume <id> — 元の cwd に chdir して claude --resume に exec する
  • ccsession（引数なし） — bash スクリプトを生成して fzf TUI を起動する

重要なのは4つめだ。fzf TUI の中では、プレビュー表示やインクリメンタル grep のたびに ccsession が自分自身をサブプロセスとして再帰的に呼び出す。 

# fzf に渡される bash スクリプトの骨格
id=$("$CCSESSION_BIN" list --color=always | fzf \
  --preview "$CCSESSION_BIN preview --query {q} {1}" \
  --bind "change:reload($CCSESSION_BIN list --grep {q})" \
  | awk -F'\t' '{print $1}')
[ -n "$id" ] && exec "$CCSESSION_BIN" resume "$id"
含　意

親プロセス（fzf を起動した ccsession）から子プロセス（reload / preview / resume で呼ばれる ccsession）へ状態を渡す経路は、環境変数しかない。 「いまどのエージェントのセッションを見ているか」という状態も、 env で運ぶしかない。これが抽象化設計の最初の制約になる。 既存コードには前例がある: --exclude-dir フラグは CCSESSION_EXCLUDE_DIR という env に変換されて子に継承される。

fzf 5列タブ契約

list が吐く TSV は、列の位置に意味が固定されている。 fzf は --with-nth=3,4,5 で後ろ3列だけを表示し、選択結果からは列1（ID）だけを取り出す。 

ID \t epoch \t 相対時刻 \t dir名 \t ラベル
└ 列1: resume に渡る  └ 列3〜5: fzf が表示・検索する

つまり、どのエージェント由来のセッションであっても、この5列に変換できれば fzf まわりの UI は一切変えずに済む。逆に、タブや改行を含む値が紛れ込めば 契約ごと壊れる。データの出どころが増えるとき、まず守るべき境界はここだ。

1.22つのエージェントのセッション保存形式

抽象化対象である2つの agent tool は、セッションの保存方法がまるで違う。

保存形式の対比
Claude CodeOpenCode（v1.2.0+）
置き場所 ~/.claude/projects/<proj>/<uuid>.jsonl ~/.local/share/opencode/opencode.db
形式 JSONL（追記型ログ、1行 = 1イベント） SQLite（WAL モード、session / session_message テーブル）
一覧の取り方 ディレクトリ走査 + 各ファイルの末尾パース（並列） SELECT ... ORDER BY time_updated DESC 1本
ID の形 UUID（= ファイル名） ses_ プレフィックスの文字列
時刻 ISO 8601 文字列 epoch ミリ秒（Date.now() 由来）
grep の単位 ファイルを開いて全行スキャン SQL の LIKE で粗く絞り、Go 側で確定マッチ
再開コマンド claude --resume <id> opencode --session <id>
観　察

「ファイル群を歩く」と「DB に問い合わせる」——実装はまったく似ていない。 しかしユーザーがやりたいことは同じだ: セッションを一覧し、中身を覗き、絞り込み、選んで再開する。 抽象化の手がかりは常に、実装の類似ではなくユースケースの同一性にある。

なお OpenCode の DB は稼働中のアプリと併読することになるため、読み取りは read-only 接続（mode=ro + busy_timeout）で行う。 また Go から SQLite を読むのに cgo を使うと、ccsession の 「CGO_ENABLED=0 で4ターゲットをクロスコンパイルする」配布パイプラインが 壊れるため、純 Go ドライバ（ncruces/go-sqlite3）を選ぶ—— 抽象化の選択は、ビルド・配布の制約からも逆算される。

1.3Go の interface と依存性逆転

Go の interface は「このメソッド群を持っていれば、それはこの型である」という 構造的型付けで、実装側は宣言不要で interface を満たす。 抽象化で使うのはこの一点だけだ。 

── Before: 呼び出し側が具象に直接依存 ──
list / preview / resume ──→ internal/session (JSONL 決め打ち)

── After: 呼び出し側は抽象にだけ依存（依存性逆転）──
list / preview / resume ──→ interface Source
                              ↑ 満たす        ↑ 満たす
                       claudeSource      opencodeSource
                       (JSONL 走査)      (SQLite SELECT)

呼び出し側（list.Run や resume.Run）は「セッションをくれる何か」 としか知らなくなり、JSONL なのか SQLite なのかは interface の向こう側に隠れる。 3つめのエージェントが来ても、呼び出し側は変更ゼロで済む——これが狙いだ。

1.4抽象化の2つの罠

罠　一 — 漏れる抽象（Leaky Abstraction）

どんな抽象も、下の実装の都合をどこかで漏らす。「漏れをゼロにする」のは不可能で、 目指すべきは漏れる場所を予見し、仕様として文書化しておくこと。 無自覚な漏れだけが事故になる（Step 4 で実例を見る）。

罠　二 — 早すぎる抽象化（Premature Abstraction）

「将来 N 個のエージェントに対応するかも」とプラグイン機構や動的レジストリを 作り込むと、2例しかないのに 10 例分の複雑さを背負う。 具体例が2つ揃った今このときが抽象化の適齢期であり、 抽象は2例が実際に要求する範囲に絞る（Step 3）。 間違った抽象より重複のほうが安い、という格言を忘れないこと。

二

抽象化の手順 — 七つのステップ

The Seven Steps

ここからが本題。ccsession の Source インターフェース設計を、 再現可能な手順に分解する。各ステップは「原則」と「ccsession での実例」の対で進む。

STEP 1

共通の「動詞」を抽出する

原　則

インターフェースは実装の共通部分からではなく、ユースケースの動詞から作る。「ファイルを読む」は動詞ではない。「セッションを一覧する」が動詞だ。

両エージェントに共通する操作を、UI が要求する順に列挙すると5つに収まる:

  • 発見する — 全セッションを新しい順に列挙（→ Scan）
  • 絞り込む — 本文を検索してマッチした集合を得る（→ GrepKeys + ScanFiltered）
  • 特定する — ID から1セッションを引く（→ FindByID）
  • 中身を見る — ヘッダ情報と直近メッセージを得る（→ Messages）
  • 再開する — 実行すべきコマンドを知る（→ ResumeSpec）

これがそのまま interface になる:

type Source interface {
    Name() string                       // "claude" | "opencode"
    Scan() ([]*session.Session, error)
    ScanFiltered(allow map[string]struct{}) ([]*session.Session, error)
    FindByID(id string) (*session.Session, error)
    GrepKeys(query string, regex bool) (map[string]struct{}, error)
    Messages(s *session.Session, max int) (Header, []Message, error)
    ResumeSpec(s *session.Session) (bin string, args []string, err error)
}

逆に、片方にしか意味のない操作は interface に入れない。 「JSONL のパスを返す」「SQL を実行する」のような実装語彙が顔を出したら、 動詞の抽出をやり直すサインだ。

STEP 2

共通データモデルを決める

原　則

新しい共通型をゼロから作る前に、既存の型がすでに「表示に必要な最小集合」を定義していないかを疑う。下流の契約（ここでは fzf 5列）が型を規定しているなら、それに乗るのが最短で安全。

ccsession の既存 Session 型は fzf 5列契約のために絞られた構造体で、 OpenCode のデータもそのまま流し込める。新フィールドは Source string の1つだけ。

OpenCode → 既存 Session 型へのマッピング
Session フィールドOpenCode 由来注意点
IDsession.idタブ・改行を含まないか検証（DB 由来値を信用しない）
CWDsession.directory空文字 → CWDUnknown → resume 拒否
Labelsession.title必ず SanitizeLabel（制御文字除去）を通す
LastEpochtime_updatedミリ秒 → 秒変換。未来時刻は沈める（既存ロジック共用）
JSONLPath—（空）OpenCode 経路はファイルに到達しない
Source（新規）"opencode" 固定resume 分岐のキー

注目すべきは 埋められないフィールドの扱いを決めていること （JSONLPath は空）と、境界でのデータ検証 （サニタイズ・ミリ秒変換）を新ソース側の責務にしていること。 共通型に流し込む時点で、データはもう「出どころ不明の外部入力」ではなく 「契約を満たした内部データ」になっていなければならない。

STEP 3

粒度を絞る — やらないことを決める

原　則

抽象は「いま具体例が2つあること」だけを根拠に設計する。共通化できる処理は interface の外側（呼び出し側）に残し、差分だけを interface の内側に置く。

好例が ResumeSpec だ。「resume を実行する」メソッドにはせず、 「実行すべきコマンドを返す」だけにしている。 

── interface の外（全ソース共通の処理）──
cwd の存在検証 → os.Chdir(s.CWD) → syscall.Exec(bin, args, env)

── interface の中（ソースごとの差分のみ）──
claude  : bin="claude",   args=["claude", "--resume",  id]
opencode: bin="opencode", args=["opencode", "--session", id]

「元のディレクトリに戻ってから再開する」という ccsession の UX 原則は エージェントに依存しない共通の価値なので、外側に1回だけ書く。 差分は「コマンド名とフラグ」というデータに圧縮された。

同じ理由で、明示的にやらないと決めたものがある:

  • プラグイン機構 — 動的ロードもレジストリも作らない。3例目が来たら struct を1つ足すだけ
  • fzf 内でのソース切替 — TUI の状態機械が複雑化する。起動時に1ソース固定
  • 既存パッケージのリネーム・移動 — internal/session はそのまま。リファクタ範囲を最小化して Claude 側のリグレッションを構造的に防ぐ
STEP 4

抽象の漏れを「仕様」で管理する

原　則

漏れは消せない。名前を間違えないことと、暗黙の挙動を明文化することで無害化する。

実例① — 「ID」という名前の嘘

grep の結果（マッチしたセッションの集合）を ScanFiltered に渡して 絞り込み一覧を作る。この集合のキーを当初 GrepIDs（ID の集合）と 名付けていたが、これは嘘だった。既存の Claude 実装はマッチ結果を JSONL ファイルパスの集合で受け渡ししているからだ。 

claude  の GrepKeys → { "/home/u/.claude/projects/a/uuid.jsonl", ... } // パス
opencode の GrepKeys → { "ses_8f2k...", ... }                       // ID

「ID の集合を返す」と契約に書いた瞬間、Claude 実装はパス→ID 変換を強いられ、 「既存コードを無変更で委譲する」という移行戦略（Step 6）と矛盾する。 解決は名前と仕様の修正だ: キーは Source 内部で閉じた opaque トークンとし、 GrepKeys → ScanFiltered の間でのみ受け渡し、外には意味を漏らさない—— と契約に明記する。漏れを塞ぐのではなく、漏れの形を正確に記述した。

実例② — 暗黙の区別を仕様に昇格させる

既存コードには「nil = フィルタなし」「空 map = マッチゼロ」という 区別が偶然成立していた。抽象化はこの暗黙の挙動を契約として明文化する 好機になる。放置すれば、2つめの実装が偶然を再現し損ねてバグになる。

実例③ — 見た目もまた契約である

プレビューのヘッダには「開始時刻がゼロ値なら最終時刻で代用」「メッセージの 時刻不明は --:-- と表示」といった細かい挙動がある。 これらは実装の癖に見えるが、ユーザーが目にする以上保存すべきセマンティクスだ。 Header / Message という共通契約に吸収し、 どのソースでも同じ見た目になることをテストで担保する。

STEP 5

切り替え機構 — flag → env → サブプロセス

原　則

切り替えは opt-in（指定がなければ完全に従来挙動）とし、状態の伝播はアーキテクチャの制約（1.1 の再帰サブプロセスモデル）に素直に従う。

ccsession                      # 従来どおり Claude Code（挙動変更ゼロ）
ccsession --source=opencode    # OpenCode セッションの fzf UI
ccsession --opencode           # 糖衣（Issue 提案との互換）

伝播の流れはこうなる:

① フラグ解析     --source=opencode
② env 化        os.Setenv("CCSESSION_SOURCE", "opencode")
③ fzf 起動      cmd.Env = os.Environ() + ...   // 子に自動継承
④ 再帰呼び出し   list / preview / resume が
                FromEnv() で同じ Source を復元する

ここにも設計判断が3つ埋まっている。 デフォルトは既存挙動に固定（後方互換）。 不正値は黙ってフォールバックせずエラー （CCSESSION_SOURCE=opencodee のような typo が無言で Claude 表示に 化けると、ユーザーは「OpenCode のセッションが消えた」と誤認する）。 矛盾指定（--opencode --source=claude）は exit 2。 切り替え機構の品質は、正常系ではなく誤入力時の挙動で決まる。

STEP 6

段階的移行 — Strangler Fig パターン

原　則

「抽象層の導入」と「新実装の追加」を同じ PR でやらない。前者は挙動変更ゼロのリファクタとして単独で出し、機械的に検証できる完了条件を付ける。

PR 分割と完了条件
PR内容検証可能な完了条件
PR1 Source 層導入 + 既存実装を委譲で包む + CI 安全網（4ターゲットビルド検証） 既存テストを1行も変えず green。全コマンド出力が変更前とバイト一致
PR2 SQLite ドライバ追加 + opencode 実装 + フラグ + テスト一式 一覧 → preview → grep → resume がエンドツーエンドで動く
PR3 README / usage 文言 / 手動 smoke チェックリスト ドキュメント整合

委譲ラッパという安全装置

PR1 では既存の grep.Filter 関数の中身を claudeSource へ移すが、 grep.Filter 自体は新実装へ委譲するだけの殻として残す。 理由は単純で、既存テストが11箇所この関数を直接呼んでいるからだ。 関数を消せば「既存テスト無修正で green」という PR1 の完了条件が成立しなくなる。 古い入口を残して中身だけ差し替える——絞め殺しの木（strangler fig）が 宿主を覆っていくように、旧構造を生かしたまま新構造に移行する。

注　意

「挙動変更ゼロ」は願望ではなく検証手順とセットで初めて意味を持つ。 既存テスト無修正 green、出力のバイト一致比較、CI でのクロスコンパイル検証—— どれも機械が判定できる。「たぶん変わってない」を人間の感覚に委ねないこと。

STEP 7

失敗の封じ込め — 新ソースの障害を伝染させない

原　則

抽象化で複数実装を束ねた瞬間、一方の障害がもう一方の機能を巻き込む経路ができる。新実装の失敗モードを列挙し、それぞれを「新実装の機能だけが劣化する」形に封じる。

  • 起動前検査（preflight） — fzf の TUI に入ってからのエラーは ユーザーに見えない（alternate screen の中で stderr が消える）。 だから DB の解決・存在検査は fzf を起動する前に行い、 失敗ならふつうの端末にエラーを出して終了する。
  • 劣化モード — メッセージが1件も読めなくても、プレビューは ヘッダ + (messages unavailable: 理由) を出す。 エラーで真っ赤になるのではなく、読めた範囲を見せる。
  • 復旧手順つきエラー — OpenCode が将来スキーマを変えて クエリが失敗したら、「DB のスキーマ版数がこの ccsession の理解より新しい/古い。 ccsession を更新するか issue を立ててほしい」と次の一手まで添えて、 OpenCode 機能だけを落とす。Claude 側は無傷で動き続ける。
  • 誘導 — ses_ で始まる ID を --source なしで resume しようとしたら「--source=opencode のことでは？」と提案する。 抽象化でモードが増えたぶん、ユーザーの迷子も増える。その分の道標を足す。

外部ツールのデータ形式に依存するということは、こちらの都合と無関係に 壊れる未来を予約するということだ。封じ込めの設計は、その未来への保険である。

四

一般化チェックリスト

Generalization Checklist

ccsession の事例を離れて、「特定ツール依存のソフトウェアを複数ツール対応に 抽象化する」あらゆる場面に使える問いとして整理する。

  1. 具体例は2つ以上あるか。 1例での抽象化は推測でしかない。2例目が来てから、2例が実際に要求する範囲だけを抽象化する。
  2. インターフェースは「動詞」で書けているか。 実装語彙（ファイル、SQL、パス）が署名に現れていたら、ユースケースからやり直す。
  3. 下流の契約が共通データモデルを既に規定していないか。 表示・出力フォーマット（ccsession なら fzf 5列）が要求する最小集合に乗れるなら、新しい共通型は不要。
  4. 共通処理は抽象の外側に残したか。 差分を「データ」（コマンド名・フラグ）まで圧縮できれば、interface のメソッドは減る。
  5. 漏れる抽象に正直な名前と仕様を与えたか。 「ID」が実はパスかもしれない。opaque にすべきものを opaque と明記し、nil と空の区別のような暗黙挙動を契約に昇格させる。
  6. 切り替えは opt-in で、誤入力時にエラーになるか。 デフォルトは完全に従来挙動。typo の無言フォールバックは「データが消えた」という誤認を生む。
  7. 「抽象層の導入」と「新実装の追加」を別 PR に分けたか。 前者は挙動変更ゼロ + 機械検証可能な完了条件（既存テスト無修正 green / 出力バイト一致）で隔離する。古い入口は委譲ラッパとして残す。
  8. 新実装の障害は新実装の機能だけを落とすか。 preflight・劣化モード・復旧手順つきエラー。外部ツールのデータ形式は、こちらの都合と無関係に変わる前提で設計する。

抽象化とは、共通点を見つけて喜ぶ作業ではない。 違いをどこに閉じ込め、漏れをどう名指しし、移行をどう検証可能にするか—— その3つの規律の別名である。
